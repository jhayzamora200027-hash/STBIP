Security headers — staging & production
-------------------------------------

This project includes two places to enforce HTTP security headers:

- Application-level: `app/Http/Middleware/SecurityHeaders.php` — applied by Laravel for responses generated by the app.
- Server-level: `docs/security/apache-security-headers.conf` — include this in your Apache site configuration to cover early responses, static assets, and redirects.

Recommended deployment steps

1. Deploy the app changes to your staging environment and clear caches:

   ```bash
   composer install --no-dev --optimize-autoloader
   php artisan config:clear
   php artisan cache:clear
   php artisan route:clear
   php artisan view:clear
   ```

2. Copy the Apache snippet to the server and enable it (example for Debian/Ubuntu):

   ```bash
   sudo a2enmod headers
   sudo cp docs/security/apache-security-headers.conf /etc/apache2/conf-available/stb-security.conf
   sudo a2enconf stb-security
   sudo systemctl reload apache2
   ```

3. Verify headers with `curl -I` against your staging URL.

Notes and cautions

- The middleware reads `config/security.php` — strict headers (COOP/COEP/CORP/HSTS) are only applied when `app()->environment()` is one of `['production','staging']` by default.
- HSTS is only set when the request is TLS (`$request->secure()`). Only enable HSTS in production when you serve all traffic over HTTPS.
- Content-Security-Policy in the Apache snippet is intentionally conservative; you should remove `'unsafe-inline'` from CSP and switch to nonces/hashes for inline scripts/styles where possible.
